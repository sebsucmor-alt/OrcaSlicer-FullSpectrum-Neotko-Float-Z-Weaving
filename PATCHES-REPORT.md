### OrcaSlicer's configuration system

Understanding how the configuration system works is critical for adding new parameters:

- **`PrintConfigDef`** — global database of all parameters (name, type, tooltip, etc.). Registered in `PrintConfig.cpp` inside `init_fff_params()`.
- **`PrintObjectConfig`**, **`PrintRegionConfig`**, **`PrintConfig`**, **`FullPrintConfig`** — static structs generated by Boost.PP macros in `PrintConfig.hpp`. Every key declared in a macro **must** have a matching entry in `PrintConfigDef` or the application crashes on startup with "Unhandled unknown exception" immediately after logging "Initializing StaticPrintConfigs".
- **`StaticPrintConfig::initialize_cache()`** — validates at runtime that every key declared in the struct macro exists in `PrintConfigDef`. If it doesn't, it throws an untyped exception that OrcaSlicer catches as a crash.
- **`Preset.cpp::s_Preset_print_options`** — list of all key strings saved in a print preset. Omitting a key here won't crash anything, but the key won't be persisted in presets.

**Golden rule when adding a new parameter:**
1. Add the key to the appropriate struct in `PrintConfig.hpp` (macro `PRINT_CONFIG_CLASS_DEFINE`)
2. Register it in `PrintConfig.cpp::init_fff_params()` with `this->add("name", type)`
3. Add it to `s_Preset_print_options` in `Preset.cpp`
4. Add the UI in `Tab.cpp`

## Features 5 and 6 — Neotko Neoweaving

This is the most complex and technically interesting feature in the build.

### Context and naming

Neoweaving was invented by Neotko (the same person who invented Ironing). It was previously called "Interlayer Sanding" in the code — a name that caused confusion with the Ironing function. It was renamed to **Neoweaving** in all user-visible labels and keys.

The internal keys remain `interlayer_neoweave_*` and `infill_neoweave_*`.

### The core idea

When printing surface layers with a monotonic pattern, adjacent lines within the same layer touch at their sides, but there's a vertical gap between layers (the layer height). Neoweaving makes lines from layer N physically nestle into the gaps of layer N-1, creating a mechanical interlock that:

- Increases resistance to delamination
- Improves surface cohesion
- Can improve visual finish with certain materials

There are two variants of the same concept:

- **Feature 5 — Surface Neoweaving:** acts on `erTopSolidInfill` (the last layer of the top surface)
- **Feature 6 — Infill Neoweaving:** acts on `erInternalInfill` (sparse internal fill)

### Configuration parameters

**Quality > Neotko Neoweaving** (top surface, Feature 5):

| Parameter | Internal key | Default | Description |
|-----------|-------------|---------|-------------|
| Enable | `interlayer_neoweave_enabled` | false | Master switch |
| Mode | `interlayer_neoweave_mode` | Wave | Wave or Linear |
| Z amplitude | `interlayer_neoweave_amplitude` | 0.10 mm | How far the nozzle moves up/down |
| Period | `interlayer_neoweave_period` | 0 (auto) | Cycle length; 0 = use line width |
| Max Z speed | `interlayer_neoweave_max_z_speed` | 20 mm/s | Z axis speed limit |
| Min line length | `interlayer_neoweave_min_length` | 3.0 mm | Shorter lines are not neoweaved |

**Strength > Neotko Infill Neoweaving** (sparse infill, Feature 6):

| Parameter | Internal key | Default | Description |
|-----------|-------------|---------|-------------|
| Per-object override | `infill_neoweave_enabled` | Inherit | Tristate (see below) |
| Z amplitude | `infill_neoweave_amplitude` | 0.10 mm | |
| Period | `infill_neoweave_period` | 0 (auto) | |
| Max Z speed | `infill_neoweave_max_z_speed` | 20 mm/s | |

**Why `infill_neoweave_enabled` is a tristate enum, not a bool:**  
If it were a `bool`, a `false` value would be indistinguishable from "the user hasn't configured anything" (no override). With the tristate `InfillNeoweaveOverride::Inherit/Enable/Disable`, per-object override can force activation or deactivation regardless of the global preset setting.

### Wave mode — Continuous wave ✅ Validated on hardware

The nozzle follows a **continuous triangular wave** along the entire path. The path is subdivided into micro-segments (minimum 8 per period) so the wave is smooth.

**Z offset calculation for each micro-segment:**

```
period = interlayer_neoweave_period  (or line_width if 0)
phase  = fmod(distance_traveled / period, 1.0) × 4   ← value 0..4

if phase < 1:  z_off = amplitude × phase           ← rising
if phase < 3:  z_off = amplitude × (2 - phase)     ← falling
else:          z_off = amplitude × (phase - 4)     ← rising again
```

The nozzle traces peaks and valleys continuously while extruding.

**XY speed cap (Wave mode):**  
For the Z axis to follow the wave without losing steps, XY speed is globally capped:

```
XY_speed_max = max_z_speed × period / (4 × amplitude)
```

This ensures the actual vertical speed never exceeds `max_z_speed`. At the end of the path, the original speed is restored.

**Important note — Wave mode needs improvement.** Known issues:

1. **Global XY speed cap:** The speed limiter applies to the entire path, unnecessarily slowing down long straight sections where Z movement is minimal. It should be calculated per micro-segment based on the actual Z angle needed at that point.
2. **Triangular vs sinusoidal wave:** The triangular wave has abrupt direction changes at peaks/valleys that can cause ringing. A sinusoidal or quadratic wave would be mechanically smoother.
3. **Pressure advance coordination:** Rapid Z changes during extrusion should coordinate with pressure advance to avoid over/under-extrusion at the wave vertices.
4. **Verbose GCode:** Subdividing into micro-segments generates very long GCode for large paths. Consider using G5 (Bézier) or G2/G3 with helical Z movement if the printer supports it.
5. **CoreXZ/CoreXZY only:** On printers with a Z-moving bed (classic Cartesians), moving Z while extruding is imprecise. The kinematic type should be detected and Wave mode automatically disabled if there's no CoreXZ.

### Linear mode — Per-line flat Z ⚠️ Implemented, awaiting hardware test

In Linear mode, **each line is printed entirely flat at a fixed Z offset** — the nozzle does not move in Z during extrusion. The height change happens in the travel between lines (pure Z move, no extrusion), not during the print move itself.

**Why this architecture:**  
On many printers (especially Cartesian), moving Z while extruding creates artifacts because the Z axis is slow or has too much inertia. With Linear mode, Z only moves during travels (no filament moving), and extrusion is always perfectly flat.

**How the zigzag between lines works:**

```
Even layers (m_layer_index % 2 == 0):
   Line 0 → Z = nominal + 0      (at nominal layer Z)
   Line 1 → Z = nominal + A      (raised A mm)
   Line 2 → Z = nominal + 0
   Line 3 → Z = nominal + A
   ...

Odd layers (m_layer_index % 2 == 1):
   Line 0 → Z = nominal + A      (starts raised)
   Line 1 → Z = nominal + 0
   Line 2 → Z = nominal + A
   ...
```

**Design change in session 7 (from ±A to 0/+A):**  
The first implementation used `nominal - A` and `nominal + A` (symmetric around nominal Z). It was changed to `nominal + 0` and `nominal + A` because with ±A, when two objects on the same bed share Z, the negative travels of one object can visually intersect with the positive lines of another, creating an interference pattern (moiré effect). With 0/+A, no lines go below the layer's nominal Z, eliminating the collision.

**Exact per-line sequence:**
1. `travel_to_z(target_z)` — pure Z move, no extrusion
2. `extrude_to_xy(end_point, dE)` — flat extrusion at that Z

**Inter-layer coordination (the heart of the system):**  
The parity of the layer index (`m_layer_index % 2`) inverts the zigzag phase. Where layer N places a line at `+A`, layer N+1 places its line at `+0` — exactly where the previous layer's gap was. Lines from consecutive layers interdigitate in Z.

```cpp
// GCode.cpp — parity logic
const bool layer_flip   = (m_layer_index % 2 != 0);
const bool line_is_even = (weave_line_idx % 2 == 0);

// XOR: if layer and line have the same parity → positive z_off
const double z_off = ((line_is_even != layer_flip)
    ? weave_amplitude   // raised line
    : 0.0);             // line at nominal Z

gcode += m_writer.travel_to_z(m_nominal_z + z_off, "Neoweaving: line Z");
gcode += m_writer.extrude_to_xy(point_to_gcode(line.b), dE, "Neoweaving linear");
```

**Critical bug fixed — arc fitting bypass (session 7):**  
All neoweaving code was inside an `if (!enable_arc_fitting || ...)` block. With arc fitting enabled (the normal case in OrcaSlicer), that block **never executed**. The only Z movement appearing in the GCode was OrcaSlicer's anti-scratch lift, not neoweave.

Fix: compute `any_neoweave` **before** the if-block and add it as a condition:

```cpp
const bool any_neoweave = weave_active_pre || infill_weave_active_pre;
if (!m_config.enable_arc_fitting
    || path.polyline.fitting_result.empty()
    || m_config.spiral_mode
    || sloped != nullptr
    || any_neoweave)      // ← forces G1 branch when neoweave is active
{
    // ... neoweaving code
}
```

**Critical bug fixed — parity bug with monotonic fill (session 7):**  
Monotonic fill is one connected polyline, not individual lines. The short connector segments between parallel lines also incremented `weave_line_idx`, inverting the parity of all long lines (all ended up at the same Z instead of alternating).

Fix: automatic filter with `max(user_min_length, 2 × line_width)`. Connectors (always < 1 line_width) are always filtered even if the user sets `min_length = 0`.

**Z restoration after each path:**
```
travel_to_z(nominal_z)    // return to exact layer Z
set_speed(F)              // restore speed (Wave mode only)
```

### erSolidInfill support in Linear mode

In session 7, `weave_active` was extended to include `erSolidInfill` (the solid infill layer immediately below the top surface) when mode is Linear. This lets that layer also participate in Z interdigitation, improving adhesion between solid infill and the surface.

**Note:** The correct enum in OrcaSlicer for internal solid infill is `erSolidInfill`, not `erInternalSolidInfill` (which does not exist).

### Solid infill angle lock for Linear mode

For Z interdigitation to work correctly, the solid infill layer below the top surface must have the same fill angle as the top surface. OrcaSlicer by default rotates the angle each layer.

**Solution:** In `FillBase.cpp`, when neoweave Linear is active, angle rotation is suppressed for `stInternalSolid` surfaces:

```cpp
// Read neoweave flags via dynamic option() lookup (no new key in struct needed)
const ConfigOption* opt_en   = print_object_config->option("interlayer_neoweave_enabled");
const ConfigOption* opt_mode = print_object_config->option("interlayer_neoweave_mode");
if (opt_en && opt_en->getBool()) {
    const auto* mode_opt = dynamic_cast<const ConfigOptionEnumGeneric*>(opt_mode);
    if (mode_opt && mode_opt->value == 1 /* Linear */)
        neoweave_lock_angle = true;
}
if (!is_using_template_angle && !neoweave_lock_angle)
    out_angle += this->_layer_angle(...);
```

**Why dynamic `option()` instead of a new key:**  
The `interlayer_neoweave_*` keys are `PrintRegionConfig`, and `FillBase.cpp` only has access to `print_object_config` (`PrintObjectConfig`). Adding a new key to `PrintObjectConfig` requires modifying the struct macro and registering the key in `PrintConfigDef` — any mismatch between `.hpp` and `.cpp` during incremental builds causes `initialize_cache()` to crash on startup. Using dynamic `option()` avoids that risk entirely.

## Feature 7 — STL file origin preservation ✅ Validated

### The problem
When importing an STL or 3MF file into OrcaSlicer, the object always appeared centered on the bed, ignoring the file's original XY coordinates. This broke workflows where multiple parts are intentionally positioned relative to each other in the source file.

### The root cause
`ModelObject::rotate()` always calls `center_around_origin()` internally, even when the rotation angle is 0 degrees. The pre-load loop in Plater called `obj->rotate(0, Z)` on every imported object — effectively calling `center_around_origin()` without any user request, destroying the XY position before `load_model_objects()` could capture it.

### The fix
```cpp
// Before: always rotated (and therefore always centered)
obj->rotate(preferred_orientation, Z_axis);

// After: only rotate if there's a real angle to apply
if (std::abs(preferred_orientation) > 1e-9)
    obj->rotate(preferred_orientation, Z_axis);
```

And auto-centering is only applied when the offset is exactly `(0.0, 0.0)` — meaning the object is genuinely at the origin and needs to be placed on the bed.

**Validated cases:** single STL, N STLs without grouping, N STLs grouped, STL whose origin is (0,0) of the model.

## Feature 8 — Temporal Link (joint movement groups) ✅ Validated

### What it does
Allows grouping objects to move together in XYZ in the viewport **without affecting slicing**. Useful for parts that have a defined positional relationship (e.g., top and bottom of a shell, a part and its custom support).

Unlike OrcaSlicer's native groups (which affect slicing), Temporal Link is purely a viewport concept — linked objects keep their independent slicing configuration.

### Architecture

**Model core:**
- `int link_group_id { 0 }` in `ModelObject` — the group ID (0 = no group)
- Serialized in cereal for in-memory persistence

**Runtime state (Plater::priv):**
- `m_link_groups` — map of group_id → list of objects
- `m_next_link_group_id` — counter for new groups
- `m_pre_move_offsets` — positions before a move (to calculate deltas)
- `m_syncing_links` — flag to prevent infinite recursion when propagating moves
- `m_floating_z_guard` — defense against the Z=0 snap bug (see Bug 10)

**3MF persistence:**  
The BBS 3MF save functions (`store_bbs_3mf`, `read_from_archive`) are in the BBS source and outside the patch set. The solution is encoding the group as a prefix in the object name: `[L1]Object Name`. On load, `rebuild_link_groups_from_model()` parses the prefixes and reconstructs the groups. Also saved as explicit metadata in the Prusa 3MF format via `3mf.cpp`.

**Re-entrancy Protection (Sync Lock):**  
In session 9, a "UI Cascading Updates" issue was identified where updating a group triggered recursive text calls (wxEVT_TEXT) when rotating or transforming. This was mitigated by injecting a boolean guard `m_syncing_group` directly into `Plater::changed_object` which nips the GUI chain reaction in the bud.

## Bug 10 — Floating objects snapped to Z=0 ✅ Fixed

### The problem
Objects legitimately floating above the bed (z > 0) were being dragged to Z=0 when moved in the 3D view.

### The cause
The snap-to-bed code in `GLCanvas3D.cpp` used:
```cpp
if (shift_z != 0.0)  // ← wrong: also "corrected" floating objects
    obj->translate(0, 0, -shift_z);
```

### The fix
```cpp
if (shift_z < 0.0)   // ← correct: only raises objects that are below the bed
    obj->translate(0, 0, -shift_z);
```

This change is applied at 4 locations in `GLCanvas3D.cpp`. The secondary defense `m_floating_z_guard` in Plater uses `wxCallAfter` to detect and revert incorrect snaps that happen outside the direct movement path.

## Feature 13 — Apply Process Preset to selected objects ✅ Validated

### What it does
A button in the top bar of the process panel (ParamsPanel) opens a combo with all **User Presets** (not system or default presets) and applies their values to the selected object as per-object overrides.

### Why only User Presets
System presets can contain values for machine parameters (multi-material, etc.) that don't make sense as per-object overrides. Filtered with `!p.is_system && !p.is_default`.

### Which keys are applied
Only keys valid as per-object overrides: `PrintObjectConfig().keys()` + `PrintRegionConfig().keys()`. Keys from `PrintConfig` (machine-level) are excluded. MM (multi-material) keys are also excluded.

**Technical note:** `Preset::print_options()` returns the full list of keys in a print preset, which is broader than valid per-object override keys. Using `print_options()` directly would try to apply invalid keys as overrides. Always use `PrintObjectConfig().keys() + PrintRegionConfig().keys()`.

## Feature 14 — Monotonic Interlayer Nesting ✅ Validated

### What it does
Shifts monotonic top-surface fill lines by half a line spacing perpendicular to the fill direction on alternate layers. Lines from layer N fall into the gaps of layer N-1, improving surface cohesion without changing Z (unlike Neoweaving).

**UI:** Strength > Top/bottom shells > "Monotonic interlayer nesting"

### Activation conditions
- `monotonic_interlayer_fill == true`
- Fill type is `FillMonotonic` (top surface pattern = Monotonic)
- Role is `erTopSolidInfill`
- `layer_id` is odd (alternates between layers)

### Implementation
In `FillBase.cpp`, the fill origin point is shifted half a spacing perpendicular to the fill direction:

```cpp
const coord_t half_sp = coord_t(scale_(this->spacing) * 0.5);
fill_dir.second += Point(
    coord_t(-std::sin(fill_dir.first) * double(half_sp)),
    coord_t( std::cos(fill_dir.first) * double(half_sp)));
```

### Synergy with Neoweaving Linear
Enabling both features produces complete 3D interdigitation:
- Feature 14 shifts lines in XY (half spacing perpendicular)
- Neoweaving Linear alternates them in Z (between nominal and +amplitude)

The result is that each line from layer N is physically surrounded by lines from layer N-1 across all three axes.

## Feature 15 — Save Object Config as Preset (session 7)

### What it does
An upload button in the top bar of the ParamsPanel opens a text dialog to enter the new preset name. The system:
1. Clones the currently active print preset as a base
2. Applies the selected object's per-object overrides on top (same safe_keys as Apply)
3. Saves the result as a new detached User Preset
4. Reloads the preset combo to reflect it immediately

### Implementation
`Plater::save_selected_object_config_as_preset(const std::string& preset_name)`:
- `prints.get_edited_preset()` gets the active preset (it's a reference, not a pointer)
- Creates a copy of the preset as `Preset tmp`
- Applies overrides from the selected object's `ModelObject::config`
- `prints.save_current_preset(name, detach=true, save_to_project=false, &tmp, nullptr)`

## Feature 16 — Process panel as right-side panel (session 8)

### What it does
Moves the process parameter panel (the fuchsia-marked area in the reference image) from the left sidebar into an independent pane on the right side of the window. The panel can be shown/hidden independently of the sidebar.

### Architecture (wxAuiManager)

1. A `right_container` (wxPanel) is created wrapping `params_panel->get_top_panel()` + the full `ParamsPanel`

3. `toggle_process_panel()` in `Plater::priv` inverts `m_process_panel_shown` and calls `pane.Show()/Hide()` + `m_aui_mgr.Update()`

### Toggle controls
- **Canvas button** — second item in the existing `collapse_toolbar` (the one containing the sidebar collapse button), using `expand.svg` icon
- **Panel's own button** — `ScalableButton` with `sidebar` icon at the far left of the ParamsPanel top bar. Allows hiding the panel even when the sidebar is collapsed.

**State persistence (Session 9):**
The visible/hidden state of the right side is now properly persisted across sessions using `wxGetApp().app_config->set("process_panel_visible")` from the `Plater` class.

## Pending work / Known improvements needed

### Neoweaving Wave — Remaining Limitations
The sinusoidal implementation mitigated most of the stuttering, but secondary constraints remain:
1. **Global XY speed cap:** The speed limit is applied to the entire path. It should be calculated per micro-segment.
2. **Pressure advance coordination:** Z trajectory changes during extrusion should coordinate with pressure advance (PA).
3. **Printers without CoreXZ:** On Z-bed printers (Cartesians), this might remain imprecise.

## Build commands

```bash
# Full build (from scratch)
./build_release_macos.sh -x

# Incremental build (reuses compiled deps)
./build_release_macos.sh -s -x
```

The ZIP files are copied over the repository source tree and compiled normally. There are no patching scripts — the ZIP files directly replace the originals in `src/`.
